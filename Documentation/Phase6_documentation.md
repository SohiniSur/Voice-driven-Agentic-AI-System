Phase 6 Documentation: Audio Feedback & Text-to-Speech (TTS) Integration
 1. Architectural Overview:
 Phase 6 closes the conversational loop by providing synchronous audio feedback to the user. While Phase 3 successfully generated text-based prompts, relying solely on visual text creates a barrier for the target demographic of this research (elderly and rural populations with potentially low digital and print literacy). By integrating a Text-to-Speech (TTS) engine, the system transforms from a text-based chatbot into a fully accessible, bi-directional voice agent.
 2. Environment & DependenciesBackend TTS Engine: 
 gTTS (Google Text-to-Speech Python library). Chosen for its ease of integration and immediate support for regional accents without requiring complex cloud authentication for the MVP.Data Encoding: Python's native base64 library.Frontend Playback: Native HTML5 Audio API (React/JavaScript).
 3. Core Concept: 
 Synchronous Base64 Audio TransportA common challenge in web-based TTS is ensuring the audio plays at the exact moment the UI updates. If the backend saves an .mp3 file and the frontend tries to fetch it via a standard URL, it can lead to caching issues or asynchronous delays.To solve this, the architecture utilizes Base64 Encoding:The backend generates the TTS .mp3 file locally.The file is immediately read and converted into a Base64 encoded string.This string is embedded directly into the master response_payload alongside the text question and the form state.The React frontend decodes the string and plays it instantly from memory, guaranteeing that the visual UI update and the audio playback happen in perfect unison.
 4. Full-Stack Integration FlowQuestion Generation: 
 The ReAct Dialogue Manager (Phase 3) determines the next question string.Audio Synthesis: FastAPI passes the string to gTTS, specifically utilizing the tld='co.in' parameter to generate the audio with a localized Indian English accent for better user familiarity.Encoding & Transport: The audio is saved, encoded to Base64, and attached to the audio_base64 key in the JSON response payload.Frontend Execution: The React sendAudioToBackend function receives the payload. Upon updating the text UI, it initializes a new Audio object with the Base64 dataURI (data:audio/mp3;base64,...) and triggers .play().
 5. Troubleshooting, Errors & Mitigations
 Error 1: Browser Autoplay Policy Blocking Initial AudioSymptom: When the React application initially loads and fetches the very first question from the /api/v1/init endpoint, the agent's voice does not play automatically. However, it plays perfectly for all subsequent questions.
 Root Cause: Modern web browsers (Chrome, Safari, Edge) enforce strict Autoplay Policies to prevent websites from playing unsolicited audio. The browser physically blocks the HTML5 <audio>.play() method until the user has meaningfully interacted with the DOM (Document Object Model), such as clicking a button.
 Resolution / Mitigation: This is an intentional security feature of the browser, not a bug in the system architecture. The system naturally resolves this constraint by requiring the user to click the "Start Recording" button to answer the first question. This initial click satisfies the browser's user-interaction requirement, cleanly unblocking the Audio API for the remainder of the conversational session. A .catch() block was added to the audio.play() promise to gracefully log the initial block without crashing the frontend application.
